# Profiler вЂ” Metrics-based profiler & benchmarker for Rust

## Project overview

A unified library for profiling, benchmarking, and production metrics in Rust.
Core idea: define metrics once (via the `Metrics` trait), reuse them for benchmarks, profiling, and production monitoring.

## Architecture

### Core layers

1. **Metrics trait** (`src/lib.rs`) вЂ” abstraction over measurable quantities (`start()` / `end()` pattern).
   - `PerfEventMetric` вЂ” wraps Linux `perf_event` counters (instructions, cache misses, task-clock, вЂ¦).
   - `InstantProvider` вЂ” wall-clock time via `std::time::Instant`.
   - Combined metrics are structs implementing `Metrics` with tuple `Start`/`Result` types (see `src/expanded_macro.rs` for the hand-expanded example; a derive macro will generate this later).

2. **Collector** (`src/lib.rs`) вЂ” a `tracing_subscriber::Layer` that snapshots metrics on span enter/exit and stores `ProfileEntry` events in a queue. Used for production-style tracing profiling.

3. **Bench** (`src/bench.rs`) вЂ” benchmark runner.
   - `Bench<M>` вЂ” benchmark builder with `.group()` / `.name()` / `.run()` API.
   - `BenchRunner<M>` вЂ” orchestrates benchmark execution, creates `Bench` instances.
   - `bench_main!` macro вЂ” generates `fn main()` that runs registered benchmarks.

4. **Report** (`src/report.rs`) вЂ” formats and displays benchmark results (mean, stddev, min, max), with optional comparison to previous runs.

## Conventions

- **Metrics are the first-class concept.** Wall time is just a metric, same as instruction count or cache misses.
- **tracing integration** вЂ” use `tracing::info_span!` / `#[tracing::instrument]` for marking regions. The `Collector` layer captures them automatically.
- **perf_event** вЂ” Linux-only perf counters are exposed through `PerfEventMetric`. Counters are per-thread (via `ThreadLocal`).
- **No allocations in hot path** вЂ” benchmark measurement loop should avoid heap allocation. The `Metrics::start()` / `end()` calls should be cheap.
- Rust edition 2024, stable toolchain.

## Adding a new metric source

1. Create a struct implementing `Metrics` (with `Start`, `Result`, `start()`, `end()`, `metric_names()`, `result_to_f64s()`).
2. Add it as a field to `MetricsProvider` (or the combined metrics struct).
3. Update `MetricsProvider`'s `Metrics` impl to include the new field in tuples and arrays.

## Style

- Keep public API minimal and composable.
- Prefer concrete types over trait objects where performance matters.
- Use `std::hint::black_box` to prevent dead-code elimination in benchmarks.
- Error handling: `unwrap()` / `expect()` is fine for metric counter setup (fail-fast on misconfiguration). No panics in the measurement hot loop.
